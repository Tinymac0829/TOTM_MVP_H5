# MVP功能文档规则

更新日期：2026-04-23

## 目的

本文档用于定义 TOTM MVP 开发过程中，功能文档应该如何编写和分级。
目标不是强迫每一次改动都写重文档，而是为了降低返工、降低误解、提高交付稳定性。

## 总原则

1. 文档的存在是为了降低返工、降低歧义、提高交付稳定性。
2. 默认优先保证项目基线文档和主 TODO ID 级文档完整，再判断子功能是否需要独立文档。
3. 只有当一个功能同时存在玩法定义风险和技术实现风险时，才需要同时写策划案和技术方案。
4. 能用一页功能卡说清的，就不要上重文档。
5. 能用验收清单说清的，就不要单独开方案文档。

## 文档分级

### L0 - 无单独文档

适用于小修、小 UI 调整、小数值微调、纯 bug 修复，或者在既有冻结规则下进行的机械实现。

要求：
- 在 thread 中说清改什么、不改什么。
- 在 thread 中说清验收口径。
- 保持简短，通常几行到一小段即可。

### L1 - 功能卡

适用于单一子功能、边界清晰、耦合较少的切片。

必填部分：
- 目标
- 范围
- 交互或规则
- 依赖
- 验收标准
- 风险

建议篇幅：约 0.5 到 1 页。

### L2 - 技术方案文档

适用于主 TODO 级功能、跨模块功能，或者需要统一实现口径的功能。

必填部分：
- 背景
- 模块边界
- 状态流
- 数据结构或配置
- 关键规则
- 边界情况
- 调试方案
- 测试点
- 性能约束

建议篇幅：约 2 到 5 页。

### L3 - 策划案 + 技术方案

适用于核心玩法、核心手感、跨 thread 协作，或者高不确定性、高返工风险的功能。

策划案应覆盖：
- 玩家目标
- 玩法规则
- 反馈表现
- 数值
- 失败与成功条件
- 验收标准

技术方案应覆盖：
- 架构
- 时序与执行顺序
- 状态机
- 模块拆分
- 数据格式
- 边界条件
- 测试矩阵
- 风险处理与替代方案

建议篇幅：约 5 到 12 页。必要时使用表格和规则矩阵。

## 升级条件

当功能满足以下任一条件时，应升级到更高一级的文档标准：

1. 影响核心手感，例如滑行、输入缓冲、速度、碰撞优先级。
2. 影响核心规则定义，例如触发顺序、死亡判定、重开行为。
3. 同时涉及两个及以上模块，例如输入、运行时、HUD、关卡格式。
4. 需要多 thread 或多 worktree 协作，不写清边界就容易做散。
5. 规则或数值尚未冻结，后续大概率反复修改。
6. 一旦做错，返工成本高，或者会污染后续很多功能。

## 降级条件

当功能满足以下任一条件时，可以保持轻量文档：

1. 只改一个模块，且不引入新规则。
2. 规则已经在上位文档中冻结，这次只是实现。
3. 回滚成本低，出错不会影响很多后续功能。
4. 验收口径可以用 5 到 8 条清单明确表达。

## 线程协作规则

### 项目进度管理 thread

- 管里程碑、状态同步、session ID 台账。
- 不为每个小功能都产出重文档。

### 任务管理 thread

- 判断这个功能属于 L1、L2 还是 L3。
- 维护该功能的唯一真值文档。

### 子功能 thread

- 消费既有文档并执行。
- 默认不主动扩写中央文档。
- 如果发现规则必须变化，再把更新回推到对应任务文档。

## 默认执行策略

1. 每个主 TODO ID 默认至少是 L2。
2. 每个子功能默认先按 L1 处理，不自动升级成双重重文档。
3. 只有核心玩法和核心手感功能，才默认按 L3 处理。

## 快速判定规则

可以按下面这条路径快速判断：

1. 这是不是一个只在冻结规则下进行的小修或机械实现？
   如果是，用 L0。
2. 这是不是一个边界清楚、耦合不高的单一子功能？
   如果是，用 L1。
3. 这是不是需要跨模块对齐，或者需要统一实现口径？
   如果是，用 L2。
4. 这是不是在定义玩法、手感，或者高风险协作边界？
   如果是，用 L3。

## 写作标准

在这个项目里，正式文档的目标是：详细到另一个 thread 只看文档，也能低歧义地实现或验收该功能，而不需要再去拼接零散聊天记录还原意图。

如果存在不确定项，必须显式标记为：
- 已核实
- 推断
- 待确认

## L1 功能卡模板

```markdown
## 功能卡：[功能名称]

**目标**：一句话说清这个功能要达成什么

**范围**：
- 包含：xxx
- 不包含：xxx

**交互/规则**：
- 规则 1
- 规则 2

**依赖**：
- 依赖模块 A 的 xxx 接口
- 依赖数据格式 B

**验收标准**：
- [ ] 验收点 1
- [ ] 验收点 2

**风险**：
- 风险 1 + 应对方案
```

## L2/L3 边界判断标准

L2 和 L3 的核心区别在于是否需要策划案。以下标准用于辅助判断：

**L2 升级到 L3 的信号**：
- 如果这个功能的“玩家体验目标”和“技术实现方案”之间存在多种可能的映射关系，比如“手感要爽”可以用不同的速度曲线、缓冲窗口、碰撞优先级来实现，就需要 L3。
- 如果这个功能的玩法定义虽然已经在上位文档中冻结，但属于核心手感或核心玩法，仍然应该按 L3 处理。要把冻结定义重写为独立的策划案 + 技术方案，并备注来源，例如“源自逆向报告中冻结的 xxx 内容”。
- 如果技术方案基本确定，只是需要明确实现细节，L2 就够了。

**保持 L2 的信号**：
- 玩法定义已经冻结，且不属于核心手感或核心玩法。
- 技术实现路径基本唯一，不存在多种可选方案。
- 不需要单独验证玩家体验目标。

## 文档保鲜期与更新规则

### L3 文档

- 一旦冻结，后续修改必须走“变更申请”流程：在文档末尾追加变更日志，记录变更原因、变更内容、影响范围。
- 变更日志格式：`| 日期 | 变更类型 | 变更内容 | 影响范围 | 审批状态 |`
- 未经变更流程的直接修改视为违规。

### L2 文档

- 允许在实现过程中迭代，但每次迭代必须更新文档头部的“最后更新日期”。
- 如果迭代导致模块边界、状态流或数据结构发生结构性变化，需要在文档末尾追加简要变更记录。

### L1 功能卡

- 如果规则变了，直接修改功能卡内容，不需要保留历史版本。
- 修改后更新日期即可。

### L0

- 不留文档，thread 中说清楚即可。
- 无保鲜期要求。

## 文档验收标准

不同级别的文档有不同的合格标准：

### L1 合格标准

- 另一个开发者看完后，能列出 5 到 8 条验收清单，且与原作者的清单 80% 一致。

### L2 合格标准

- 另一个开发者看完后，能画出模块交互图和状态流，且与原作者的理解 90% 一致。

### L3 合格标准

- 另一个开发者看完后，能独立实现出手感接近的原型，且玩家体验目标达成。
- 策划案部分：另一个策划看完后，能准确复述玩法规则、失败/成功条件、数值基线。
- 技术方案部分：另一个开发者看完后，能准确复述架构、时序、状态机、数据格式。

## 文档存放规则

### L3 文档

- 存放目录：`docs/design/`
- 文件名格式：`<功能名>_design.md`（策划案与技术方案合并为一个文件，内部分章节）
- 如果策划案和技术方案篇幅都很大，允许拆分为 `<功能名>_gdd.md`（策划案）和 `<功能名>_tech.md`（技术方案），但必须互相引用。

### L2 文档

- 存放目录：`docs/tech/`
- 文件名格式：`<功能名>_tech.md`

### L1 功能卡

- 存放目录：`docs/features/`
- 文件名格式：`<功能名>_card.md`
- 也可以直接放在对应 task worktree 的根目录，视项目规模决定。

### L0

- 不留文档文件，thread 记录即可。

## 当前项目的建议映射

### 默认 L3

- PM-02 技术设计文档（核心运行时、网格格式、状态流、输入、碰撞、关卡格式、HUD、调试、部署）
- ENG-04 核心移动手感（四向滑行、输入缓冲、碰撞处理、速度与压力曲线）

### 默认 L2

- 关卡格式与关卡加载器
- HUD 状态流
- 调试面板
- 重开与通关流程

### 通常 L1，必要时升级到 L2 或 L3

- 单个机关或机制接入
- 如果规则已经在主规则表里冻结，用 L1。
- 如果规则还没冻结，升级到 L2。
- 如果它会改变玩法定义或手感，升级到 L3。

### 默认 L1

- LVL-01 Story 1 接入
- LVL-02 Story 2 接入
- LVL-03 Story 3 接入

### 默认 L0 或 L1

- 按钮样式调整
- 文案更新
- 纯展示层调整
- 冻结规则下的已知 bug 修复

## 实际目标

目标不是让文档数量最大化。
目标是用最低必要的文档级别，保证执行稳定、口径统一、过程可复核。
